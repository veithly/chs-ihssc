# 提交材料：价序

初赛材料匿名。赛道：医药价格治理。

## 项目名称

价序

## 一句话

价序是给医保医药价格治理岗的对话式智能体工作台：把对不齐的价格表交给它，用一句话布置工作，它跑完一个可回放的 agent 循环，把能处置的项写成 5 类可审批对象。

## 项目摘要

医保医药价格治理岗每天面对的是一堆对不齐的表：省平台挂网价、机构回传的执行价、集采中选价落地进度、昨日催过的回函，常常混在一起；包装单位写错、目录别名没对上，又让一条线索看起来像违规。工作人员先要决定今天处理哪几件、哪些要补证、哪些只能观察。直接看一张大表，只会把人拖回筛选和判断。

价序把这个动作做成一个对话式工作台。首屏 `/` 是 landing：产品定位、实时工作台统计、业务 prompt rail（hero prompt 主打政策变更后的存量执行价复核闭环）、产品循环图。用户点一个 prompt，深链进 `/workspace?prompt=&text=`，演示数据源自动 attach。`/workspace` 是现代 agent 工作台：顶 band 显示 agent 标识、线程、provider 和状态；左侧是对话区和输入框；右侧是数据上下文 + 业务对象面板（漂移队列 / 人审任务 / 处置建议卡 / 规则候选 / 政策事实 / 数据修复）+ 审计日志条。

用户发送指令后，服务端调用真实模型，智能体先读字段、归并同品、换算单位、按规则评估，再由确定性服务端工具把结果写成 5 类可审批对象——字段映射、修复 patch、同品归并、流程任务、机构核实草稿——全部写入本地 SQLite，过程落 run_event。对话框给出一份交代（处理多少行、做了什么），右侧面板自动切到第一个非空 tab。拿不准的项不硬写：缺字段的不催医院，可疑但不致命的先转人工，并生成 needs_user 追问。用户的 follow-up 指令（如"把重点机构排前面"）会触发重排，任务和草稿随之更新。模型负责出计划，状态、人审边界和恢复由服务端管；发函、通报、违规认定、关闭线索仍由业务人员确认。Provider 失败时返回 auth_failed，确定性对象仍可见，草稿标为不可用，绝不伪造结论。

针对一线"政策实时更新数据跟不上、审批负担重"的反馈，V2.2 增加政策对齐与自学习闭环：政策事实（policy_fact）版本化入库作为 baseline 真相源，支持从国家医保局公开公告页真实抓取、公告 artifact 人审确认后才更新；每次 run 后对照 baseline 检出存量执行价的政策漂移（policy_drift_log），高危漂移自动生成「政策漂移复核」人审任务。人审批准时记录实际处置动作（final_action），沉淀进不可变决策日志；规则引擎从决策日志按（问题类型 × 严重度 × 处置动作）挖掘规则候选（support/confidence 门槛，来源决策 id 可逐条审计），dry-run 预览影响面后由人工激活；下批 run 命中激活规则且护栏通过的项自动处置，高危/超最高有效价/集采超容忍/编码失效等敏感项被护栏永久留在人审。自动与人审决策（needs_human / human_approved / auto_approved / ratify_rule）全部留痕可回放。

项目覆盖医药价格数据标化治理、药品及医用耗材价格动态监测、价格异常预警与处置、集采价格落地跟踪。当前演示使用合成/脱敏数据，成功路径调用真实模型；不声称已接入真实省级招采平台、HIS 或生产工单。

## 对照评审维度

### 场景契合度（25%）

首屏就是价格治理岗的一天：landing 上点一句话，进工作台，把杂乱价格表推进到可审批的处置对象。字段、对象和状态围绕价格目录、中选价、执行价、集采落地、证据缺口和机构口径。

### 技术可行性（25%）

关键路径调用 OpenAI 兼容的真实 provider（国产 step 模型）。agent 编排 observe→plan→tools→mutate→verify(+drift/learn) 阶段，每步留回放。服务端负责状态写入和人审边界：漂移检测对照版本化 policy_fact 现读 baseline，学习规则自动处置前先过确定性护栏，LLM 不拍板状态结论。SQLite 保存会话、数据集、5 类对象、流程任务、政策事实、漂移记录、规则候选、决策日志和运行记录。Provider 失败时写失败态，不生成假对象。

### 产品完整度（20%）

评委可完成一条完整路径：打开 `/`，点 prompt，进 `/workspace`，看 agent run，看 5 类对象，回 follow-up 触发重排，看工具轨迹回放；失败态可见 auth_failed。桌面和移动端都能看核心流程。线程可重开，对象持久。

### 创新性（15%）

价序没有把价格治理做成大屏、按钮墙或审计表。它让智能体承担"读表、归并、换算、评估、写处置对象"的脏活，把判断和边界留给人；用户用自然语言追问就能改变整条处置链。更进一步：人审结论不是一次性的——同模式的一致人审会被挖掘成规则候选，人工激活后下批自动处置，审批负担随使用递减，而敏感红线由护栏保证永远不被自动化。

### 应用潜力（15%）

后续可接省级招采平台、机构执行价回传、采购订单进度、投诉回函和工单系统，并扩展到基金监管、经办服务、数据赋能等相邻赛道。当前演示不声称这些生产系统已接入。

## 演示路径

1. 打开 `/`，看实时工作台统计；点 hero prompt `核完并闭环处置这批机构执行价异常`，自动进 `/workspace`，agent 当场跑。
2. 右侧「政策事实」tab 点政策变更演示（640→560）或政策同步 + 人审确认，重跑 hero prompt。
3. 「漂移队列」看检出的存量执行价漂移；高危漂移已自动生成「政策漂移复核」人审任务。
4. 「人审任务」批准同类项并选处置动作；「规则候选」挖掘 → dry-run → 人审激活。
5. 再跑一次：命中规则且护栏通过的项「自动处置」，敏感项仍留人审；审计日志条全留痕。
6. 回 needs_user 追问（follow-up），任务/草稿重排；看工具轨迹回放。
