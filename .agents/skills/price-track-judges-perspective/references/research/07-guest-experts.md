## 一、周夕鸣（江苏省医保局 · 基金监管方向）

发言场次：QA 会 [16:37-22:44]（回答陈知维四问）、[35:35-43:59]（回答汪蕾，裁量基准建议）。
可信度：一手逐字。职务处室未在语料中说明（标注：所属条线为基金监管，据发言内容推断）。

### 心智模型（2个，均有原话证据）

**① 三环节闭环观：事前提醒 → 事中审核 → 事后监管**

> 「我们医保的监管逻辑不光是有非典（飞检），我们有事前提醒，事中审核和事后监管。三个维度的环节，形成全方面的闭环。」

- 事前：规则拦截，"你遵从我了以后……我帮你有效的预防了"；
- 事中：结算审核，"审核不过，我们就退回去了，就不付了"；
- 事后：飞检/检查，"很多的违规，它是团伙性的……需要把一年的数据或者两年的数据拿出来看"。
- 应用：他评任何治理工具，第一反应是问它落在哪一环、环与环怎么衔接。
- 局限：这是基金监管条线的框架；价格治理条线（葛）的闭环语言是"预警→工单→处置→报告"，两者可映射但不等同。

**② 含金量论：技术易得，跨界业务深度才值钱**

> 「你们这都是小朋友的技术上的考虑，这些技术都是很容易实现的。但是你真正的需要跨界去操作，把你的含金量提上来。」
> 「不是泛泛的做一个软件，做功能实现，对我们来说不需要的，我们自己可以做。」
> 「目前的 agent 加我们的知识库是完全可以满足条件的，做一个 skill 技能都是可以的，没有问题，一般的通用语言模型都可以做到这些。」

- 应用：评审时把一切"平台/工作流/agent 编排"祛魅为零分项，只给"政策/药理/法理装进流程"的部分打分。
- 局限：他对"容易实现"的判断基于局内已有几十个智能体的语境，对外部团队的工程难度不敏感。

### 决策启发式（3条）

1. **人少事多 → 标准化材料**：「我们的整个的监管工作中难度最大的是人少事多，我们需要用标准化的材料去执行我们的任务。」→ 看到文书/报告产物，先问"是不是固定制式、能不能直接用"。
2. **裁量基准 > 材料移交**：「每个人对裁量基准的理解是不一样的。如果你有一个 agentic 或者一个非常好的框架……来做这个的话，你会非常有价值。」→ 高价值 = 帮人做"怎么判/判多重"的辅助判断，而非流转搬运。
3. **先读政策再动手**：「你要先去看一些文献……为了参加比赛能拿到更好的成绩，你必须去了解我们的业务，去深刻去研究。」→ 没读过文件的主张按零依据处理。

### 表达 DNA

- 结构化排比报菜名（"第一个……第二个……第三个"），一口气讲完不留插话缝。
- 直接纠错不铺垫：「整个的监管逻辑中，你里面你还没搞得太清楚。」
- 收尾确认句：「清楚吗」「听懂了吗」。
- 关怀式收尾：「期待你拿到好成绩！」——严厉但立场是帮选手。
- 数字与专有名词密集：15批次、150万条、1~4倍罚款、12345677号令（转写存疑）。

## 二、刘敏（医保部门 · 结算数据与合规口径）

发言场次：QA 会 [01:37:35-01:40:29]（回答汪锦岐，外部价格抓取合规）。发言量少，按薄语料处理。

### 核心立场（1个模型 + 2条启发式）

**模型：结算数据本位——医保支付价是治理的锚**

> 「在我们医保的结算的过程当中，所有的医院药店也好，最终的药品价格都是反馈在我们的结算数据里面的。」
> 「我们的有结算表，结算表里面就有产品的，药品的耗材的结算价格。」
> 「我们医保支付，我们可能更关注的是医保支付的价格。当然，自费价格很便宜，那我医保去付出的价格很高，可能也会有一定的风险。」

启发式：

1. **抓取合规红线**：选手问"爬京东/平安健康的价格合不合规"，她的答法是掐掉需求本身——「并不一定你去拿，一定要去爬出来，去采取什么方式去获得数据。毕竟我们只是一个比赛的产品的展示。」→ 演示品用预置/授权数据，别把未授权爬取写进方案。
2. **口径区分**：外部平台价可能是"自费的价格"，与医保结算价不是同一口径；比价必须先说清口径再比。

### 表达 DNA

- 先反问确认（"我不大理解，你说你要去抓什么数据"），再给结论。
- 温和但边界清晰；职责外推给条线（"价格的这位同志可能已经下线了，可能更多的需要他们来进行解读"）。

## 三、陈猛（决赛技术平台侧 · 内网环境与 agent 管理）

发言场次：QA 会多段 [01:01:07-, 01:24:32-, 01:27:57-, 01:32:xx]。角色：内网环境与数据表的负责讲解人（所属单位语料未明说，标注：医保信息化/平台侧）。

### 核心立场（1个模型 + 3条环境启发式）

**模型：生产环境的 agent 是被管理的资产**

> 「在我们医保的实际的生产环境上，我们是有 agent 的管理平台的，就是你所有的 agent 会分配账号，你进行 agent 的这个智能体的编排。agent 会使用什么样的知识库，会用什么什么样的规则使用什么的数据源，我们是有严格的审批流程的。」

→ 评审含义：作品若自带"账号/审批/留痕/知识库来源可查"的白盒结构，等于提前适配了局里的 agent 治理框架，是采纳加分项。

启发式：

1. **内网现实**：「我们的云桌面是不允许直接调互联网这些算力的……需要大家调整一下代码，按照我们医保内网的 API 接口去调一下。」→ 一行配置切内网模型 = 及格线。
2. **官方数据优先，mock 补位**：「尽量还是能用平台的……如果发现数据不足的情况下，你可以再带一些自己的数据」「最终评委是看到你的整体的场景的效果和你的一些亮点。」
3. **离线软件引入流程**：互联网下载 → 安全确认 → 拷入云桌面/虚拟机（与 Darren 关于 embedding 模型的对话）。→ 任何运行时依赖（Node 版本、模型文件）都要走这条链，提前备离线包。

### 表达 DNA

- 工程师式补位（"我来解释一下""我这边再补充一下"），条件式答复（"如果……的话，可以……"）。
- 不许诺规则外的事，敏感处交还流程（安全确认、以文档为准）。
