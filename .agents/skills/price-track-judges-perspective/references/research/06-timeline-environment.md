## 决赛技术环境事实卡（评审"能不能现场跑"用）

- 24 队，4 赛道 × 6 队；两个奖项（具体设置见选手指南，语料未含）。
- 开发：自带电脑自由开发，工具/模型不限（vibe coding 不影响评分）。
- 测试与路演：**医保内网云桌面**（登录自己账号），数据只进不出，签保密协议。
- 算力：云桌面 + 每队虚拟机 8C16G、存储约200G，可装 MySQL/PG，可申请扩容。
- 模型：**仅内网私有化千问 235B**，OpenAI 风格 chat/completions API，统一 token（24队共用），文档给 URL/请求头/参数/示例；**不允许云桌面调外网**。
- 无内网 embedding 模型；需要向量化 → 外网下载模型文件 → 安全确认 → U盘/审核通道拷入 → CPU 跑。
- 数据：40 张脱敏表 × 约1000条（医保结算参保 / 门急诊住院诊疗 / 检查检验 / 院内记录 四大类）；**结算费用明细表含药品耗材结算价格**；表头真实、数值脱敏；可自带 mock 数据补场景，但"尽量用我们准备的数据"。
- 路演当天：内网盒子搬到摊位现场，产品在盒子上跑给评委看。

## 对演示可行性评审的直接推论

1. 任何依赖外网 API（模型、抓取、CDN、字体、npm 运行时安装）的关键路径，现场都可能断——必须有离线降级或提前拷入。
2. 模型调用必须能一行配置切到内网千问（base_url + token + model 名）。
3. 演示数据要提前想好"官方40张表能撑哪段、自带 mock 撑哪段"，并能向评委说清这个区分。
4. 政策同步类"真实抓取"功能在内网必然失败——要么演示离线政策包导入，要么明确标注该功能为外网能力并优雅降级。
