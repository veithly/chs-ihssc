# 价序 · 游园会演示脚本 + 评委应答卡

> 场景：7月5日游园会摆摊制，无路演无 PPT，评委和专业观众走到摊位前看真东西。
> 评委构成：省医保局价格招采线（葛文婧）、基金监管线（周夕鸣）、结算数据线（刘敏）、信息化线（陈猛、卢银桂处长）、省数据集团（周超、黄乐平）。
> 核心策略：先让评委 30 秒内认出「这是给我们价格治理岗用的」，再用一条完整闭环打透，最后用政策文号接住追问。

---

## 一、30 秒开场（评委走近时）

「老师好。这是价序，给医保价格治理岗用的智能体工作台。它做一件事：把治理岗每天面对的乱价格表——编码对不上、单位不一致、集采没落地、价格超限——交给智能体先跑一遍，能按规则处置的自动处置，拿不准的和敏感的留给人审，每一步可回放可审计。您可以直接点，我给您演示一条从政策变更到自动处置的完整闭环。」

要点：
- 第一句话就说清用户是谁（价格治理岗），呼应行知强调的「用户画像明确」。
- 不说「大而全平台」，说「插进已有系统的一个环节」——葛文婧原话是异常数据报告、工单闭环这些「细节点」。

---

## 二、5 分钟标准演示动线

| 步骤 | 操作 | 讲词（关键句） |
|------|------|---------------|
| 1. Landing | 打开 `/`，指实时统计 | 「这些数字是数据库里的真实状态，不是静态图。」 |
| 2. 上传核价 | `/workspace` 上传 `settlement-sample.csv`，指令「核完并闭环处置这批机构执行价异常」 | 「这份表模拟结算明细：编码残缺、单位不一致、别名。智能体先读字段做映射，**对齐国家医保编码**，归并同品同规，再按规则评估。」 |
| 3. 看差比价 | 指向阿莫西林 48 粒装的核验任务 | 「同通用名不同包装不能直接比价。按**发改价格〔2011〕2452号**差比价规则，包装数量翻倍价格上限乘 1.95：24 粒代表品 8.72 元，48 粒上限就是 17 元，这行报 19.8 超限，进人审核验——督促企业调价是敏感动作，护栏不允许自动发出。」 |
| 4. 批次闸门 | `/release/REL-2026-0623-07` 启动扫描，看结果页 | 「整批 42 行一键扫描。超最高有效价的按**苏医保发〔2021〕64号**分档打标：1.3 倍是黄色预警提醒关注，10 倍以上红色预警三星，暂停交易资格。1.3 倍零售比价用的是**国办发〔2026〕9号**全渠道比价口径——零售价没有医保编码，系统先做名称映射再折算比价。」 |
| 5. 政策闭环 | 「政策事实」tab → `政策变更演示 640→560` → 重跑 | 「人工晶体中选价从 640 调到 560（**医保办函〔2026〕38号**接续采购公示）。政策确认后重跑，存量执行价漂移全部检出，高危自动生成复核任务——这就是『政策实时更新、数据跟不上』的解法：政策事实版本化，每次 run 对照最新基线。」 |
| 6. 自学习 | 人审批准同类项 → 规则候选挖掘 → dry-run → 激活 → 再跑 | 「人审结论沉淀成规则：support/confidence 达标才生成候选，dry-run 预览影响面，人审激活后下批非敏感项自动处置。**审批负担随使用递减**，但高危项被护栏永久留在人审。」 |
| 7. 收尾 | 打开任一「处置结果单」+ 审计日志条 | 「每条线索从发现到处置生成结果单；自动和人审决策全部写不可变决策日志——监管闭环要求的『处置完能收集回来』，在这里。」 |

时间不够时的 2 分钟精简版：步骤 2 → 4 → 5（上传核价 → 分档预警 → 政策漂移），够展示「标化、监测、闭环」三个赛题关键词。

## 三、评委应答卡（按人备）

### 葛文婧（价格招采，最可能追问细节）

**Q：差比价怎么算的？**
「按发改价格〔2011〕2452号第十三条包装数量差比价：K = 1.95^log₂X，X 是包装数量比。演示里 48 粒对 24 粒，X=2，K=1.95，代表品 8.72 元 × 1.95 = 17 元上限。规格差比价（含量维度）用系数 1.7，同一公式，目录里补 comparableGroup 就能扩展。」

**Q：零售价没有医保编码怎么对应？**
「这正是我们做的点之一。零售渠道行走名称+规格模糊映射到目录，映射置信度低的行不硬比，标『零售价无编码可对应』进数据治理确认——宁可交给人，不出假异常。」

**Q：1.3 倍规则的口径？**
「国办发〔2026〕9号的全渠道比价要求：挂网/采购价不高于当地零售药店及网售平台『即时达』价格集中区间的 1.3 倍。我们取零售集中价 × 1.3 做上限，超限行进需核验，不直接判违规。」

**Q：工单闭环怎么收回来？**
「每个流程任务有 owner 角色和状态机，处置动作（final_action）落决策日志，处置结果单聚合每条线索的证据链。批量下发后回收看『工作台统计 + 审计日志』，不用一条条翻。」

**Q：监测指标怎么深度优化？**
「两层：规则层跑 R1-R7（已含 64 号分档、9 号文比价、2452 号差比价）；学习层从人审决策挖掘新规则候选，dry-run 影响面人审激活——指标是人定的，但候选是从处置行为里长出来的，这就是『让它更聪明』的路径。」

### 周夕鸣（基金监管，考监管逻辑理解）

**Q：你们这个在监管三环节里算什么？**
「事前和事中之间的价格侧防线。挂网前批次闸门拦截是事前提醒；日常监测发现存量漂移是事中督导。我们不做事后飞检——但决策日志和处置结果单可以给事后检查提供留痕证据。」

**Q：智能体乱写数据怎么办？**
「智能体只写工作状态：映射、归并、草稿、任务。发函、通报、违规认定必须人确认。护栏是服务端硬编码，模型 prompt 越狱也绕不过——敏感项类型直接写死在 `guardrails.ts`。」

### 刘敏（结算数据）

**Q：数据从哪来？**
「演示数据全部合成/脱敏，字段对齐大赛 40 张表里的结算明细结构。真实落地时接结算表——您说过药品耗材结算价格都在结算表里，我们的 schema mapper 就是为『字段名不一致的结算表』设计的，40 张表哪张字段变了都能映射。」

### 陈猛 / 卢银桂（信息化、内网环境）

**Q：内网跑得起来吗？**
「跑得起来，这台就是按内网标准配的。全栈 Node + SQLite 单文件，无外网依赖；大模型调用走 OpenAI 兼容接口，`OPENAI_BASE_URL` 指到内网千问 235B 就行。兼容层已处理 response_format 不支持的情况；模型超时 90 秒自动降级写失败态，不假成功。部署清单在 `docs/DEPLOYMENT_OFFLINE.md`，8C16G 云桌面 10 分钟起。」

**Q：白盒审计怎么做的？（呼应赵可同学问题，陈猛关心）**
「三层：run_event 记每步工具调用的输入摘要和结果；replay_timeline 记 observe→plan→tools→mutate→verify 全过程；approval_decision_log 不可变记录每个自动/人审决策和依据。任何一条处置结论都能从结果单反查到是哪次 run 哪个工具哪条规则得出的。」

**Q：跟我们已有的几十个智能体怎么共存？**
「价序是小切口：不替换现有系统，输入是导出的 CSV/结算表，输出是处置建议和流程任务。已有 agent 管理平台的话，我们按贵局的账号/审批流注册进去，知识库和规则库用贵局的。」

### 省数据集团（周超、黄乐平）

**Q：可信数据空间里怎么部署？**
「云桌面 + 虚拟机模式完全够：项目 tar 包 + Node 运行时 U 盘导入，数据从底座同步到虚机 MySQL/PG 或直接 CSV 上传。『只进不出』没问题——我们本来就不出数据，全部落本地 SQLite。」

## 四、故障兜底话术

| 状况 | 操作 | 话术 |
|------|------|------|
| 千问超时/降级 | 指着「检查失败」状态 | 「这正是设计的一部分：provider 超时，系统写降级 run，状态诚实置为检查失败，不会拿本地规则包装成 AI 成功——医保场景假成功比真失败危险。我重跑一次。」（重跑即恢复） |
| 数据被上一轮演示改乱 | `curl -X POST localhost:3000/api/admin/reseed` | 「我把演示数据重置一下，10 秒。」 |
| 被问没做的功能 | 不硬编 | 「这个当前版本没做。我们的取舍是先把价格标化→监测→预警→闭环这条线做完整，XX 在架构上是加一个 validator/一张表的事。」 |
| 被问真实数据效果 | 诚实 | 「演示数据是合成的，但字段结构对齐大赛 40 张表。规则全部来自公开政策文号，接真实结算表主要是 schema mapper 重新对一次字段。」 |
